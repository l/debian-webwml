Debian Policy Manual - chapitre 2
Documentation

2.1 Pages de manuel

Vous devez installer les pages de manuel au format source nroff, à l'emplacement approprié sous /usr/man. Vous devez seulement utiliser les section 1 à 9 (voir le FSSTND pour plus de détails). Vous ne devez pas installer de pages préformatées `cat page'.

Si aucune page de manuel n'est disponible pour un programme, un utilitaire ou une fonction particulière et qu'un bug est reporté sur debian-bugs, un lien symbolique de la page de manuel en question vers la page de manuel 7() doit être fournit. Ce lien symbolique peut être créé à partir de debian/rules comme suit : 

   ln -s ../man7/undocumented.7.gz \
    debian/tmp/usr/man/man[1-9]/the_requested_manpage.[1-9].gz
Cette page de manuel indique que l'absence d'une page de manuel a déjà été signalée comme un bug, aussi vous avez seulement à le faire que si c'est le cas (vous pouvez le reporter vous-même si vous le souhaitez). Ne fermez pas le rapport du bug tant qu'une page de manuel adéquate n'est pas disponible.

Vous pouvez envoyer une requête concernant l'absence d'une page de manuel aux auteurs originaux, et marquer le bug comme transféré dans le système Debian de traques des bugs. Cependant le projet GNU ne considère pas en général l'absence d'une page de manuel comme un bug, nous oui - s'ils vous répondent qu'ils ne le considèrent pas comme un bug, vous devez cependant laisser le bug ouvert dans notre système Debian de traques des bugs.

Les pages de manuel doivent être installées compressées via gzip -9.

Si une page de manuel doit être accessible via différents noms, il est préférable d'utiliser un lien symbolique plutôt que la fonctionnalité .so, mais il n'est pas nécessaire de bricoler les parties incriminées dans les sources originaux pour changer les .so en liens symboliques - à moins que ce soit trivial. Ne créez pas de lien physique dans les répertoires des pages de manuel et ne mettez pas de chemin absolu dans les directives .so. Le nom du fichier dans le .so d'une page de manuel doit être relatif à la racine des pages de manuel (usuellement /usr/man).

2.2 Documents Info

Les documents Info devraient être installés dans /usr/info. Ils devraient être compressés avec gzip -9.

Votre paquet doit appelé install-info pour mettre à jour le fichier dir dans son script de post-installation : 

   install-info --quiet --section Development Development \
     /usr/info/foobar.info

C'est une bonne idée de spécifier une section pour l'emplacement de votre programme ; cela se fait avec l'option --section. Pour déterminer la section à utiliser, vous devriez consulter /usr/info/dir sur votre système et choisir la plus adéquat (ou créer une nouvelle section si aucunes des sections courantes ne sont adaptées). Notez que l'option --section prends deux arguments ; le premier est une expression régulière pour rechercher (insensible à la casse ;-) une section existante, le second est utilisé lorsqu'une nouvelle section est à créer.

Vous devez retirer les entrées dans le script de pré-désintallation : 

   install-info --quiet --remove /usr/info/foobar.info

Si install-info ne peut trouvé une entrée description dans le fichier Info, vous aurez à en fournir une. Voir 8() pour plus de détails.

2.3 Documentations additionnelles

Toute documentation additionnelle fournie avec le programme peut être installée à la discrétion du responsable du paquet. Les documents texte devraient être placés dans le répertoire /usr/doc/package, où package est le nom du paquet et compressés avec gzip -9 à moins qu'ils soient petits.

Si un paquet contient une large documentation que la majorité des utilisateurs du paquet ne requièrent pas, vous devriez créer un paquet sépare la contenant, ainsi elle ne prendra pas d'espace disque sur les machines des utilisateurs ne la nécessitant pas ou ne voulant pas l'installer.

Il est souvent une bonne idée de mettre les fichiers d'informations textuelles (README, changelogs, et autres) provenant du paquet source dans /usr/doc/package au sien des paquets binaires. Bien entendu, vous n'avez pas besoin de fournir les instructions pour la création et l'installation du logiciel.

2.4 Formats de documentation préférés

L'unification de la documentation Debian se fait autour d'HTML.

Si votre paquet fourni une vaste documentation dans un markup format qui puisse être convertible en d'autres formats, vous devriez si possible livrer la version HTML dans le paquet binaire, dans le répertoire /usr/doc/package ou un de ces sous-repertoires. [1]

D'autres formats comme PostScript peuvent être fournis en option.

2.5 Informations sur les Copyright

Chaque paquet doit être accompagné avec une copie textuelle de son copyright ainsi que de sa licence de distribution dans le fichier /usr/doc/package/copyright. Ce fichier ne doit ni être compressé ni être un lien symbolique.

De plus, le fichier copyright doit préciser où les fichiers sources originaux (s'ils existent) ont été obtenu et expliquer brièvement quelles modifications ont été faite dans la version Debian du paquet par rapport à l'original. IL doit indiquer les auteurs originaux du paquet et le(s) responsable(s) Debian qui ont oeuvré à sa création.

/usr/doc/package peut être un lien symbolique vers un répertoire de /usr/doc seulement si les deux paquets proviennent de la même source et que le premier paquet possède une relation "Depends" avec le second. Ces règles sont importantes car les copyrights doivent être ectractible par des moyens automatiques.

Les paquets distribués sous la licence UCB BSD, la licence Artistic, la GNU GPL ou la GNU LGPL doivent faire référence aux fichiers /usr/doc/copyright/BSD, /usr/doc/copyright/Artistic, /usr/doc/copyright/GPL ou /usr/doc/copyright/LGPL.

N'utiliser pas le fichier copyright comme un fichier README général. Si votre paquet a un tel fichier, il devrait être installé dans /usr/doc/package/README ou README.Debian ou une autre place appropriée.

2.6 Exemples

Tous les exemples (configuration, fichiers source, autres) doivent être installés dans un répertoire /usr/doc/package/examples. Ces fichiers ne doivent pas être utilisés par un quelconque programme -- ils sont ici pour le bénéfice de l'administrateur système et des utilisateurs comme documentation seulement.

2.7 Fichiers Changelog

Ces fichiers doivent contenir une copie du fichier debian/changelog de votre racine source Debian, et une copie du fichier changelog source s'il existe. Le fichier debian/changelog devrait être installé dans /usr/doc/package comme changelog.Debian.gz. Si le changelog source est un fichier texte, il doit être accessible à /usr/doc/package/changelog.gz. Si le changelog source est un fichier HTML, il doit être accessible à /usr/doc/package/changelog.html.gz. Si le changelog source n'est pas déjà conforme à cette convention de nommage, alors cela doit être réalisé soit en supprimant ce fichier ou en ajoutant des liens symboliques, à la discrétion du responsable du paquet.

Tout deux doivent être installés compressés par gzip -9, car ils deviendront imposants avec le temps même s'ils débutent petit.

Si un paquet a seulement un changelog qui est utilisé à la fois comme changelog Debian et source car il n'y a pas de responsable original séparé alors ce changelog devrait habituellement être installé sous /usr/doc/package/changelog.gz. S'il existe un responsable principal mais pas de changelog source alors le changelog Debian devrait être toujours appelé changelog.Debian.gz.
